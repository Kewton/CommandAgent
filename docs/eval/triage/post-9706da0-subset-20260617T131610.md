The step still failed because the exact grep verifier returned non-zero before repair, then repair attempted a compound shell command and was blocked. This suggests the remaining problem is not artifact creation but verifier/repair robustness.

### Remaining Rust modify issue

Generated inspect plan:

```yaml
- id: inspect-lib-structure
  kind: inspect
  instruction: Read src/lib.rs if it exists to identify existing module declarations and public API boundaries.
  expected_paths:
    - src/lib.rs
  verify:
    - test -f src/lib.rs
```

The instruction says "if it exists", but `expected_paths` and `test -f` make it mandatory. This is a DSL contract gap: optional inspection targets need a representation that does not become a hard gate, or the prompt/lint must reject "if exists" plus `expected_paths`.

### Remaining Rust new issue

Generated source contains:

```rust
#[derive(Parser, Debug)]
```

Generated verifier expects exactly:

```text

```

This is the same brittle source-grep family seen before. The prompt says source-code behavior should prefer build/test/check, but the planner still emits exact grep for source semantics.

## Conclusions

`9706da0` did what it was supposed to do:

- `cargo check` is no longer policy-blocked.
- `true` no-op verifier is no longer accepted in the corrected plan path.

The subset still fails because new residuals are exposed:

1. Optional inspection paths are being encoded as mandatory `expected_paths`.
2. Source semantics are still sometimes checked with brittle exact grep.
3. Repair turns still attempt compound shell instead of simple tool calls after verifier failure.

## Recommended Next Fixes

Keep the next work narrow and common:

1. Plan lint/prompt: reject or correct `expected_paths` for instructions containing optional language such as `if it exists`; use `verify: []` and direct Read/Glob inspection instead.
2. Plan lint/correction: flag exact source-code grep for semantic constructs when a build/check command exists in the same profile. Do not ban grep for docs/data literal checks.
3. Repair prompt: after verifier failure, explicitly require one simple tool command at a time and forbid shell fallback/chaining. This should be phrased as a tool-use contract, not a larger repair loop.

Do not address yet in this slice:

- Next.js dependency install/preseed policy
- Python import isolation
- larger repair budgets or sidecar summarization
