## Detailed Findings

### 1. The strongest common bug is premature final-artifact enforcement

CommandAgent currently injects `required_artifacts` into each phase step plan and
then checks those artifacts at the end of every step plan:

```text
src/agent/step_runner/runtime.rs:256-275
```

This means an inspect-only phase can fail because a final artifact planned for a
later phase does not exist yet. The `large-rust-app-modify` run shows this
exactly: the ultra plan contains later phases to create/extract modules, but the
first `analyze-structure` step plan exits with missing `src/lib.rs` before those
phases can run.

Anvil does not enforce the final artifact list at this point. Its ultra runner
runs each phase, verifies profile invariants after the phase, and continues:

```text
src/agent/minimal_step_runner.rs:580-622
```

Anvil still surfaces required artifacts in prompts, but it does not treat them
as phase-local expected paths. This is the most important CommandAgent/Anvil
delta because it was introduced while restoring R1-R4. The intent was correct:
carry the eval artifact contract forward. The implementation made it too eager.

Judgment: common contract/runtime issue. Fix before profile tuning.

### 2. Verifier command grammar is still too syntax-naive

Two failures are caused by verifier commands that are local checks in intent but
are rejected or fail for overly literal reasons:

- `large-fastapi-app-new`: `python -c "import ast; ast.parse(open('app/main.py').read())"`
- `large-nextjs-app-new`: `grep '"next"' package.json`

The Python case contains semicolons inside quoted Python source. The shell policy
sees this as command chaining even though the check is a single local Python
process. The Next.js case uses plain `grep`, while the allow-list is narrower
than the planner's natural verifier vocabulary.

This is not a request to broadly allow arbitrary shell. The common contract
should either:

1. make the planner generate simpler canonical checks, such as `python -m
   py_compile app/main.py`, `python -m pytest`, `test -f`, or `grep -q`; or
2. make the lint/policy parser shell-quote-aware enough to avoid false compound
   detections inside quoted arguments.

Judgment: common verifier/tool contract issue, with Python and Next.js symptoms.

### 3. Next.js failures are now mostly dependency/profile contract issues

`large-nextjs-app-modify` created the requested UI artifact but failed because
`npm run build` requires `node_modules/.bin/next`, and dependency installation is
blocked by policy. The repair prompt records the right evidence:

```text
dependency_missing: verifier_unavailable: npm run build requires node_modules/.bin/next
```

This is not a parser failure and not an artifact-path failure. It is a contract
question: in large eval, do we allow dependency setup, preseed dependencies, or
count dependency absence as `dependency_missing` rather than implementation
failure?

Anvil's Next.js profile had a richer contract around this case. It explicitly
said new apps need `next/react/react-dom`, build scripts must remain `next
build`, dependency setup must be a separate setup step, and missing dependencies
must not be faked as success. CommandAgent's current profile text is intentionally
smaller:

```text
CommandAgent: src/agent/step_runner/profiles.rs:58-62
Anvil:       src/agent/minimal_step_runner/profiles/nextjs.rs:6-35
```

The CommandAgent direction should remain small, but the dependency policy must
be explicit. Without that, Next.js large eval is measuring environment setup
rather than implementation ability.

Judgment: profile/eval dependency contract. Handle after the common artifact and
verifier-command fixes.

### 4. Python modify exposes import isolation, not missing artifacts

The FastAPI modify run produced `app/main.py`, `app/schemas.py`, and
`tests/test_app.py`, but pytest imported `app.schemas` from:

```text
/Users/maenokota/share/work/github_kewton/MySwiftAgent/expertAgent/app/schemas/__init__.py
```

That means the test environment resolved a different `app` package before the
workspace's own `app/schemas.py`. This is a Python package/import hygiene issue.
It can be solved by profile/env contract, for example:

- require `app/__init__.py` for package-style apps, or
- run pytest with workspace-first `PYTHONPATH`, or
- prefer a project layout that avoids ambiguous top-level `app` namespace
  packages.

This is not a reason to add a new runtime mechanism. The verifier evidence is
already clear.

Judgment: Python profile/env-specific.

### 5. Rust new shows generated verifier brittleness

The Rust new run created `Cargo.toml` and `src/main.rs`. The code contains:

```rust
#[derive(Parser, Debug)]
```

The generated verifier expected the exact string:

```text
grep -q "#\[derive(Parser)\]" src/main.rs
```

That verifier fails even though the code uses `clap::Parser` correctly. This is
a common problem with planner-generated exact text verifiers. For code tasks,
verifiers should prefer semantic local checks (`cargo check`, `cargo test`,
`python -m pytest`, `npm run build`) or broad file existence checks. Exact grep
should be reserved for literal documentation/content requirements.

Judgment: common verifier contract issue, not Rust-profile-specific.

## Anvil Delta And Why It Happened

The differences from Anvil are not accidental model behavior alone. They come
from migration choices:

1. CommandAgent intentionally removed legacy Anvil mechanisms and kept only the
   current minimal/plan-run surface. This is correct for MVP scope, but it also
   removed several contracts that had accumulated around `/ultra-plan-run`.
2. R1-R4 restored common concepts (`intent`, `step.kind`, `expected_result`,
   `required_artifacts`, workspace-aware lint), but not every Anvil behavior was
   restored exactly.
3. The most important restoration bug is treating final `required_artifacts` as
   mandatory after every generated phase step plan. Anvil used those artifacts as
   pressure and final completion context, not as phase-local success criteria.
4. CommandAgent profiles are deliberately smaller. That keeps the architecture
   maintainable, but it means framework-specific contracts such as Next.js
   dependency setup and Python package import hygiene are not yet explicit enough
   for large eval.
5. CommandAgent's Bash/verifier policy is stricter and less syntax-aware than
   the planner's verifier vocabulary. This creates false blocks for local checks
   and brittle grep checks.

The root cause is therefore not “CommandAgent lacks a bigger control system.”
It is that a few remaining contracts are applied at the wrong layer or with the
wrong timing.

## Common Vs Profile-Specific Split

Common contract/runtime issues to fix first:

1. Final `required_artifacts` must not be enforced at the end of every phase step
   plan. Keep them in prompts and enforce only at the appropriate final boundary,
   or only when a step explicitly lists them in `expected_paths`.
2. Verifier command grammar must be canonical and predictable. Generated checks
   should avoid shell chaining and fragile exact grep; lint should not misread
   quoted Python code as shell control flow.
3. Read-only verification affordances should include the planner's common local
   checks or normalize them into the supported subset.
4. Generated verifier quality should favor semantic commands over exact string
   matches for source code.

Profile/eval-specific issues to handle after the above:

1. Next.js: decide dependency setup policy for `npm run build` under offline eval
   and keep the honest build-script contract.
2. Next.js: keep port 3011 and app entrypoint contracts explicit when the case
   asks for a runnable app.
3. Python/FastAPI: make package import resolution deterministic, likely by
   package marker or workspace-first test execution.
4. Rust: after the common final-artifact timing fix, re-evaluate whether the
   Rust modify case still needs any Rust-specific module contract.

## Recommendation

Do not jump to R5 profile expansion or R6 repair observability yet.

The next migration-scoped fix should be:

1. Correct final artifact enforcement timing.
2. Tighten/canonicalize verifier command contracts.
3. Re-run the same large eval root shape.
4. Only then decide which residual failures are genuinely profile-specific.

This keeps the work aligned with the CommandAgent philosophy: reduce ambiguity
and make deterministic contracts explicit before adding larger control loops or
profile-specific rules.
