```yaml

  kind: "inspect"
  instruction: "Glob the components directory to check if AnalyticsPanel.tsx already exists or if there are existing component patterns to follow."
  expected_paths:
  verify:
    - "test -d components"
```

Failure:

```text
command: test -d components
reason: command_failed:1
repair turn: minimal loop reached max iterations
```

This is not the earlier `if exists` + `expected_paths` bug, but it is the same family: optional inspection was encoded as a fatal verifier. For inspect steps, absence of an optional directory should usually be an observation, not a failed step.

### Rust Modify

Rejected plan included shell chaining / fallback verifier syntax, and the run stopped with:

```text
plan lint failed: step `create-errors` has invalid verifier command `cargo check 2>&1 | grep -v "warning" || true # Ensure it compiles, ignoring warnings if any`: verifier commands must be one simple local check; split shell chaining into separate commands
```

This confirms the simple-command lint is active. The remaining issue is planner recovery: after a lint rejection, the next plan should be regenerated under the same constraints instead of ending the case.

### Rust New

The run produced Rust source and then rejected the following verifier:


Failure:

```text

```

This is the desired failure mode for the new lint. The previous path could fail or pass on text shape rather than Rust behavior.

## What Remains Before Returning To R5/R6

The remaining failures are still common-contract issues, so it is premature to move to profile-specific R5 or repair-observability R6.

Recommended next narrow fixes:

1. Inspect-step verifier semantics: for `kind: inspect`, missing optional directories/files should not be expressed as fatal verifier commands. Either steer inspect steps to `verify: []` unless the inspected path is truly required, or lint `test -d/test -f` when the instruction is discovery-oriented.
2. Invalid-plan bounded retry: when plan lint rejects a generated step plan, feed the lint error back to the planner once with the saved invalid plan and regenerate. This is a syntax/contract correction, not an execution repair loop.
3. Positive verifier contract examples: show replacements for rejected source-grep checks, e.g. `cargo check`, `cargo test`, `npm run build`, or a small parser/test command where appropriate.

Do not address yet in this slice:

- framework-specific profile expansion
- larger repair budgets
- semantic sidecar summarization
- additional automatic repair mechanisms

## Decision

Do not return to the original recovery plan yet. The next step should stay in the common layer and address invalid-plan recovery plus inspect-step verifier semantics. Once these no longer dominate the subset, rerun the same three cases and only then decide whether remaining failures are profile-specific or repair-observability issues.
