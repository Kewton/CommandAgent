## Detailed Findings

### 1. Plan lint false-positive candidates

Affected:

- `large-fastapi-app-new`
- `large-rust-app-new`

Evidence:

```text
ERROR: plan lint failed: step `create-pyproject-toml` mixes setup/editing work with verification
ERROR: plan lint failed: step `add-file-io-tests` mixes setup/editing work with verification
```

Assessment:

- `large-rust-app-new` is likely a productive run stopped late by lint. The
  workspace contains `Cargo.toml`, `Cargo.lock`, `src/main.rs`, and build
  artifacts under `target/`.
- `large-fastapi-app-new` failed before creating user artifacts. The invalid
  generated step plan was not saved, so this root cannot prove whether the lint
  was a false positive or a correct rejection.
- This is primarily a migration-quality concern because the same strict lint
  that protects small plans may be too coarse for generated large-task plans.

No change is made here. A later scoped fix should first save invalid generated
plans for triage, then decide whether the setup/verify lint is too broad.

### 2. Blocked inspection/install commands

Affected:

- `large-fastapi-app-modify`
- `large-nextjs-app-modify`
- `large-nextjs-app-new`

Evidence:

```text
find app/routes -name "*.py" -exec cat {} \;
cat app/layout.tsx || cat app/layout.ts
test -f node_modules/.bin/next && echo "success" || echo "missing"
npm install
```

Assessment:

- `find ... -exec cat` and `cat A || cat B` are read-only in intent, but they
  are compound shell forms. The current Bash policy blocks them unless they are
  exactly `cd <workspace> && <local check>`.
- The more migration-aligned direction is not to broadly whitelist these shell
  forms. The planner should prefer `Glob`/`Grep`/`Read` or separate verifier
  commands for inspection.
- `npm install` is different. It is dependency installation and can touch the
  network. Blocking it is consistent with the offline/default-safe policy. The
  benchmark must explicitly decide whether large Next.js eval is allowed to
  install dependencies or should stop as `dependency_missing`.

No policy change is made here.

## Anvil Parity And Root Cause

The failure classes are not all new. Anvil had already exposed several of the
same large-task problems, but later Anvil-side methodological fixes were not all
migrated into CommandAgent's MVP slice.

| Failure class | Seen in Anvil? | CommandAgent-specific root cause |
| --- | --- | --- |
| Plan lint false-positive candidates | Yes. The Anvil large report says plan lint was still too broad and rejected build steps even when setup files existed. | CommandAgent's `src/agent/step_runner/plan_lint.rs` is plan-text-only. Anvil's `src/agent/minimal_step_runner/plan_lint.rs` has `lint_plan_with_workspace(plan, Some(cwd))`, tracks introduced paths, checks existing workspace files, and has richer verification-only handling. |
| `expected_paths` glob / alternative path contracts | Partly. Anvil saw artifact path drift such as `app/components/...` vs `components/...`, but later added required final artifact propagation from `success_check.files`. | CommandAgent eval runner does not append `expected_artifacts` / `success_check` paths into the `/ultra-plan-run` prompt. Generated plans therefore invented `app/routes/*.py` and `app/layout.ts` alternatives. CommandAgent also currently saves only valid plans, so invalid linted plans are not preserved for analysis. |
| Read-only inspection blocked by Bash policy | Related issues existed. Anvil had `mkdir` and `cd ... && python/node` affordance traps, then fixed cd-wrapper handling. The specific `cat A || cat B` / `find ... -exec cat` pattern is not called out in the Anvil large report. | CommandAgent's `src/tools/bash.rs` deliberately blocks compound shell unless it is exactly `cd <workspace> && <local check>`. The generated plan used shell fallback forms for inspection instead of `Read`/`Glob`/separate simple commands. |
| Next.js dependency install / offline policy | Yes in principle. Anvil offline mode rejects env setup, but Anvil classifies `npm install` as `EnvSetup` with a dedicated message and permits it when not offline. | CommandAgent always classifies `npm install` as `Network` and the eval runner always executes without an explicit online dependency policy. The case asks for `npm run build` but the environment has no `node_modules`, so the benchmark setup and policy are inconsistent unless dependency setup is explicitly allowed or treated as `dependency_missing`. |
| Rust modify `max_iterations` stop | Yes. Anvil large report says several steps reached `minimal loop reached max_iterations (8)` and that step-level stop reasons needed better recording. | CommandAgent retains the small step cap but lacks Anvil's richer step `kind`, `expected_result`, profile snapshot, artifact contract pressure, and detailed stop-reason accounting. In this root, `define-module-tree` had a concrete expected artifact but no file-changing repair succeeded. |

The strongest CommandAgent-specific bug is intent loss. The eval cases mark
modify cases with `intent: modify`, but the runner constructs only:

```text
/ultra-plan-run --profile <profile> --style <style> <prompt>
```

and `src/agent/step_runner/runtime.rs::generate_ultra_plan` calls:

```text
ultra_plan_generation_prompt(goal, profile, style, "new")
```

As a result, saved ultra plans for modify cases contain `intent: "new"`. Anvil
did not do this: `src/agent/minimal_step_runner.rs::generate_ultra_plan`
detects intent from the goal and overwrites the generated plan's intent with
that detected value. Anvil also passes the intent into profile-specific
generation/runtime contracts.

This explains why CommandAgent modify cases drifted toward generic discovery or
new-project style plans instead of preserving and editing the existing fixture.

## Recommended Next Work

Keep the next work migration-scoped:

1. Preserve invalid generated plan text when plan lint fails.
   This is observability, not performance tuning. It is needed to distinguish
   real mixed setup/verify steps from lint false positives.
2. Restore intent handling for `/ultra-plan-run`.
   This is migration parity, not performance tuning: pass or infer
   new/modify/investigate/docs/data intent and keep it immutable in generated
   ultra plans and profile contracts.
3. Revisit large-case plan contracts so inspection steps do not put globs or
   alternate paths in `expected_paths`.
   Concrete artifact paths should remain concrete; exploratory reads should use
   `verify` or tool instructions, not expected artifacts.
4. Carry required final artifact paths from eval cases into the run prompt.
   This mirrors Anvil's `ULTRA_ARTIFACT_CONTRACT=success_check.files` behavior
   and reduces path drift without adding a new model-control mechanism.
5. Decide the eval dependency policy explicitly.
   For Next.js large cases, either allow dependency installation in the eval
   environment or classify dependency absence as `dependency_missing` without
   counting it as an implementation failure.
6. Do not broaden Bash policy as a quick fix.
   If read-only inspection needs to improve, prefer planner/tool-catalog
   affordance toward `Read`, `Glob`, and `Grep` before allowing more compound
   shell syntax.
7. Treat `large-rust-app-modify` as the max-iteration probe.
   Do not tune it yet; first ensure stop reasons and failed step artifacts are
   consistently preserved.

## Current Judgment

The `20260617T003924` root shows a migration parity concern, not a clean model
capability failure. The highest-confidence migration gaps are:

- insufficient preservation of invalid plan text on lint failure;
- overly loose generated plan contracts around `expected_paths`;
- eval dependency policy not explicit enough for large Next.js;
- large-step max-iteration behavior needs better observability before tuning.

No success-rate improvement should be attempted until these observation and
contract issues are addressed.
