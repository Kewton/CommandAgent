## Problem Statement

CommandAgent currently has several adjacent pieces:

- semantic failure reports can preserve conflict inputs;
- active-job dispatch can stop on `contract_conflict`;
- target admission can report target conflict reasons;
- verifier diagnostics can expose observed/expected pairs, affected cases,
  and weak verifier reasons.

The missing C33 contract is the deterministic decision layer that turns these
inputs into one of:

- source implementation repair;
- test alignment repair;
- docs/API/spec repair;
- verifier contract correction;
- explicit stop because authority is ambiguous.

Without C33, target priority or generic source repair can still become the
fallback when the correct behavior is to preserve the conflict and stop or
repair the authoritative side.

## Non-goals

- Do not add hidden confirmation, advisory memory, or semantic sidecar
  judgment.
- Do not let profiles become workflow engines or own the conflict decision.
- Do not weaken verifier commands, tests, docs, or API contracts to force a
  pass.
- Do not add provider/model-specific conflict handling.
- Do not increase retry budgets or continue after conflict safe stop.
- Do not implement Phase29 language/profile/tool support expansion.

## Design Alignment

Phase28 follows the current CommandAgent contract stack:

```text
TaskContract / BehaviorObligation / ArtifactGraph / ArtifactLedger
  -> SemanticFailureReport conflict inputs
  -> Phase27 target / attempt / no-progress evidence
  -> Phase28 ContractConflictJob
  -> authority decision or explicit conflict stop
  -> bounded recovery task or original verifier/guard rerun
```

The minimal loop remains the executor. Phase28 adds a deterministic conflict
contract and eval-visible fields around the existing repair path. It does not
become another repair engine.

## Contract Conflict Model

Add a small shared C33 model:

| concept | meaning |
| --- | --- |
| conflict object | Structured record of the conflicting sides, artifacts, observed/expected pair, source of evidence, and affected case. |
| authority input | Deterministic source that can justify one side: user request, behavior obligation, profile contract, existing docs/API/schema, pre-existing test, generated test binding, verifier command, or observed implementation. |
| authority decision | `implementation_authoritative`, `test_authoritative`, `docs_or_api_authoritative`, `verifier_contract_limited`, `ambiguous_authority`, or `insufficient_evidence`. |
| selected conflict action | `edit_source_for_diagnostic`, `align_test_and_verifier`, `update_docs_literal`, `replace_invalid_verifier_command`, or `stop_with_structured_evidence`. |
| safe stop | Terminal report when authority is ambiguous or insufficient. It must include conflict sides, authority candidates, missing evidence, and rerun authority. |

Authority and repair target are separate. The authoritative side is normally
preserved; the selected action repairs or rejects the non-authoritative side.
For example, a generated test without binding should not make source repair
authoritative. A pre-existing bound test can make source repair appropriate.

## Source-of-Truth Rules

The authority decision must be deterministic and evidence-bound:

1. Explicit user request or admitted behavior obligation outranks generated
   artifacts.
2. Profile contract and declared docs/API/schema can be authoritative only when
   they are bound to the current task or existing workspace artifact.
3. Pre-existing tests can be authoritative when they are in scope and not
   generated by the current failed repair attempt.
4. Generated tests are not authoritative unless they bind to an admitted user,
   profile, docs, API, or schema obligation.
5. Verifier commands are authority only for the behavior they actually check;
   weak/self-referential verifier findings must route to verifier contract
   correction or explicit stop.
6. Existing implementation can be a preservation constraint, not a default
   source of truth, unless the task contract explicitly says to preserve it.
7. Equal or missing authority must stop with structured conflict evidence
   instead of choosing by path order or target priority.

## Architecture Notes

Implementation should be shaped as a narrow contract layer:

- Prefer a new shared module such as `contract_conflict.rs` for the conflict
  object and authority decision.
- Feed it from existing producers: `task_contract.rs`, `semantic_failure.rs`,
  `recovery_orchestration.rs`, `target_admission.rs`, `artifact_ledger.rs`,
  and `verifier_diagnostic.rs`.
- Project the result into `RecoveryTaskContract`, repair packets, and eval
  fields.
- Keep action selection in the existing recovery orchestration/action envelope
  boundary. The conflict job should decide authority and selected conflict
  action, not run tools.

The selected conflict action must name both:

- the authoritative side to preserve; and
- the target side/action to repair or stop.
